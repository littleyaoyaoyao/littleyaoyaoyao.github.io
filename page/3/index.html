SVN

标签 ： SVN

版本控制（Revision Control）：是维护工程蓝图的标准做法，能追踪工程蓝图从诞生一直到定案的过程。是一种记录若干文件内容变化，以便将来查阅特定版本修订情况的系统。

创建仓库

在硬盘中创建一个空的文件夹，如F:\repository\svn\repo1
在cmd窗口中可以完成创建仓库操作，包括在指定目录创建仓库和当前目录创建仓库。
指定目录创建：svnadmin create F:\repository\svn\repo1
当前目录创建：svnadmin create .

启动仓库

  • 多仓库
    cmd > svnserve -d -r F:\repository\svn
    要访问某一特定的仓库：svn://localhost:3690/repo1

  • 单仓库
    cmd > svnserve -d -r F:\repository\svn\repo1
    此时只能访问该已经启动的仓库repo1：svn://localhost:3690

操作

checkout

svn checkout svn://localhost:3690/repo1

commit

commit需要有权限，需要修改仓库目录中conf\svnserve.conf的配置

  • 将# anon-access = read修改为anon-access = write
  • 使用add子命令添加到本地版本库

然后执行语句：
svn commit -m "user1 add content" test.txt
其中，语句中包括了-m设置日志信息

update

在每次提交版本之前先执行更新操作
svn update

目录规范

  • trunk 主线 ， 用于存在程序整个进度
  • branches 分支 ， 例如bug修复、特殊功能
  • tags 标签（版本） ， 此目录下的内容不能修改

一.spring事务管理

1.1 事务

  • 事务：一组业务操作ABCD，要么全部成功，要么全部不成功。

  • 特性：ACID
    原子性：整体
    一致性：完成
    隔离性：并发
    持久性：结果

  • 隔离问题：
    脏读：一个事务读到另一个事务没有提交的数据
    不可重复读：一个事务读到另一个事务已提交的数据（update）
    虚读(幻读)：一个事务读到另一个事务已提交的数据（insert）

  • 隔离级别：
    read uncommitted：读未提交。存在3个问题
    read committed：读已提交。解决脏读，存在2个问题
    repeatable read：可重复读。解决：脏读、不可重复读，存在1个问题。
    serializable ：串行化。都解决，单事务。

Hibernate 二级缓存

标签 ： Hibernate 二级缓存

二级缓存

二级缓存是进程级别的缓存，将数据库中的一部分数据存在缓存中，可以减少对数据库的访问。
由于数据库的数据存在硬盘中，而二级缓存是在内存中保存，并且存储速度更快。

二级缓存共有四个缓存区

  • 类缓存区
  • 集合缓存区
  • 查询缓存区 : HQL
  • 时间戳缓存区

使用进程缓存的好处

  • 缓存是高速的，缓存性能效率远远比访问数据库高
  • 访问缓存不需要建立连接，节约带宽和网络资源

缓存中的数据

  • 经常使用
  • 不经常被修改

使用二级缓存

  1. 导包

配置二级缓存的实现类

private static SessionFactory sf;
static{
    //1.加载配置
    Configuration conf = new Configuration().configure();
    //2.创建sessionFactory;
    SessionFactory sf = conf.buildSessionFactory();
    Runtime.getRuntime().addShutdownHook(new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("虚拟机关闭!释放资源");
                sf.close();
            }
        }));
    }

加载策略

延迟加载：等到使用的时候才会加载数据
立即加载：不管使用不使用，都会立刻将数据加载

类级别的加载策略

  1. get/load
  • get：立即查询数据库，将数据初始化
  • load：hbm文件中，class元素的lazy属性绝对类级别load方法的加载策略
    true : 先返回一个代理对象。使用代理对象的属性时，采取查询数据库
false : 与get一致，会理解加载数据

关联级别加载策略

在查询有关联关系的数据时，加载一方的数据是否需要将另一方理解查询出。

默认：关联的数据在使用时才会加载

一对多

  1. 使用lazy——-对于set

    <set name="orders" lazy="true">

  • true（默认值） : 是否对set数据使用懒加载
  • false : 集合将会被立即加载
  1. 使用fetch —— 对于set
    决定加载集合使用的sql语句
  • select(默认值) : 普通select查询
  • join : 表链接语句查询集合数据
  • subselect : 使用子查询加载集合数据,一次加载多个Customer订单
  1. 共六种情况排列组合
  • select + true : 默认值，会在使用集合时加载，普通slelect语句
  • select + false : 立刻使用selelct语句加载集合数据
  • join + true : 查询集合时使用表链接查询，会立刻加载集合数据
  • join + false : 查询集合时使用表链接查询，会立刻加载集合数据
  • subselect + true : 会在使用集合时记在，子查询语句
  • subselect + false : 会在查询用户时，立即使用子查询加载客户的订单数据
  1. extra 极其懒惰
    在使用集合的时候，没有使用集合的数据而是使用了集合的长度size，那么Hibernate会发送count查询数量而不会加载集合内数据。
多对一

此时应该配置在Order一方many-to-one中

  1. lazy
  • flase : 加载订单时，会立刻加载客户
  • proxy : 看客户对象的类加载策略来决定
  • no-proxy : 不做研究
  1. fetch
  • select : (默认值)使用普通select加载
  • join : 使用表链接加载数据
  1. 排列组合
  • select + proxy : 类加载策略为:lazy=false 同上
  • select + false : 加载订单时,立即加载客户数据.普通select语句加载客户.
  • join + proxy : 使用表链接查询订单以及对应客户信息.lazy属性无效
  • join + false : 使用表链接查询订单以及对应客户信息.lazy属性无效
